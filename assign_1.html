What is the Post Correspondence Problem?

Introduced by Emil Post in 1946, PCP is a puzzle about matching strings using a set of domino tiles. It looks simple — but no algorithm can solve it in general.

The Setup

You're given a finite set of dominoes. Each domino has a string on top and a string on bottom. Your goal: pick a sequence of dominoes (repetition allowed) so that the string formed by reading all the tops equals the string formed by reading all the bottoms.

Rules

  • You can use each domino multiple times
  • The sequence can be any length
  • Order in the sequence matters
  • A "solution" means top-concat = bottom-concat

Why it's tricky

  • No bound on solution length
  • You can't just check all finite sequences
  • Some instances have no solution — but how do you know when to stop?
  • That's exactly the undecidability!

Why Can't an Algorithm Solve PCP?

You might think: just write a program that tries all sequences. If a match exists, it'll find it eventually. The problem is the no case — how do you know when to give up?

The halting problem in disguise

PCP is undecidable because we can encode any Turing Machine computation as a PCP instance. If you could solve PCP, you could solve the Halting Problem — which is impossible.

The key insight — search trees grow forever

With k dominoes, at sequence length n you have kn sequences to check. For 3 dominoes:

no bound!

But the real problem isn't speed — it's that we can't even know whether the answer exists at all. There's no computable upper bound on solution length.

Semi-decidable vs. Undecidable

PCP is semi-decidable

If a match exists, a systematic search will eventually find it. We can enumerate all sequences by length and check each one. We'll halt with "YES".

PCP is NOT decidable

If no match exists, we'll search forever and never halt. We can't recognize the "NO" case. A decider must halt on every input — we can't guarantee that.

The formal statement

Theorem (Post 1946): The language PCP = { ⟨P⟩ | P is a PCP instance with a match } is undecidable but Turing-recognizable.

Proof sketch: We show ATM ≤m PCP — a mapping reduction from the acceptance problem for Turing Machines. Since ATM is undecidable, so is PCP.

Reducing ATM to PCP

We prove PCP is undecidable by showing: if we could decide PCP, we could decide whether any Turing Machine accepts any input — which we know is impossible.

The trick: encode computation histories as dominoes. A PCP match exists exactly when the TM has an accepting run on w.

What is a computation history?

A computation history is the sequence of configurations the TM passes through: tape contents, head position, and current state at each step.

Domino construction

Given TM M and input w, we build PCP dominoes that encode the step-by-step computation:

Start domino

Forces the match to begin with the initial configuration of M on w.

#q₀w#
#

Top already has the first configuration. Bottom is empty except for the separator. This forces a "head start" — MPCP variant.

Transition dominoes

For each TM transition δ(q, a) = (r, b, L/R), we create a domino that copies the tape while applying the transition:

qa
br
move right
cqa
rcb
move left (c = any symbol)

These dominoes "advance" the bottom by one step relative to the top.

Copy dominoes

For every tape symbol a, a domino [a/a] copies unchanged tape cells.

a
b
b
#
#

Accept domino

When the TM reaches an accepting state, dominoes "eat up" all remaining tape symbols and close the match with #.

qacca
qacc
aqacc
qacc
qacc##
#

Why does this work?

The top string accumulates configurations C₁#C₂#C₃#…. The bottom string accumulates #C₁#C₂#…Cₙ — always one step behind. A complete match forces C₁#C₂#…#Cₙ = #C₁#…#Cₙ#, which means the top and bottom strings can only equalize if the machine reaches an accepting configuration Cₙ that can be "absorbed" by accept dominoes.

Deep Dive — Consequences & Variants

PCP is not just an academic curiosity. Its undecidability is a powerful tool for proving other problems undecidable.

MPCP — Modified PCP

The Modified PCP (MPCP) fixes the first domino in the sequence. It's also undecidable, and the TM-to-PCP reduction naturally targets MPCP (the start domino is "first"). Standard PCP is then reduced from MPCP via a padding gadget: new symbols are introduced to simulate the start-domino constraint without fixing the first element.

PCP over binary alphabet

Even restricting to the alphabet {0, 1}, PCP remains undecidable. The TM alphabet can always be encoded in binary, so the reduction still works.

Problems that reduce FROM PCP

Because PCP is undecidable, we can use it to prove other problems undecidable by showing reductions PCP ≤ₘ X:

  • CFL Ambiguity — Is a context-free grammar ambiguous?
  • CFG Intersection — Do two context-free languages intersect?
  • Post Canonical Systems
  • Wang Tiling — Can a set of tiles tile the plane?
  • Mortality of Matrix Products

Common pitfalls

"Dominoes can be reused" — most forgotten rule

In PCP, repetition is essential — it's what makes the search infinite. Without repetition, you'd only need to check finitely many sequences (all permutations), which would be decidable.

"The strings just need to be the same length"

Wrong. The actual strings — their characters, not just their lengths — must match. The reduction encodes TM transitions so that "tops = bottoms" means "transitions are valid and the TM accepts."

"We can just run the solver and check if it loops"

This is the core misconception. We cannot detect looping from the outside — that would solve the Halting Problem. We can enumerate matches for the YES case, but there's no mechanical way to confirm the NO case.
